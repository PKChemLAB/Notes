# 第1章 算法分析
---

## 一、算法的定义
【定义】**算法**是指令的有限集合，遵循这些指令可以完成特定的任务。此外，所有算法都必须满足以下5条核心准则：

1.  **输入**：有0个或多个由外部提供的量
   
2.  **输出**：至少产生一个输出量
   
3.  **确定性**：每条指令都清晰、无歧义
   
4.  **有穷性**：对于算法的所有用例，在执行有限步之后必然终止
   
5.  **可行性**：每条指令都必须足够基础，原则上可以由人仅用纸笔完成；仅满足确定性是不够的，操作还必须是可实现的

> 补充说明：
> 
> 程序是用某种编程语言编写的，且不要求必须有穷（例如操作系统，是持续运行的程序而非算法）。
> 
> 算法可以通过自然语言、流程图、编程语言或伪代码来描述。

### 示例：选择排序
**功能**：将 n ≥ 1 个整数按升序排列
**核心逻辑**：在当前未排序的整数中，找到最小值，将其放到已排序序列的下一个位置。

#### 伪代码实现
```c
for ( i = 0; i < n; i++) {
    // 检查list[i]到list[n-1]，找到最小值的下标min
    交换 list[i] 和 list[min];
}
```
排序核心动作 = 查找未排序区间的最小值 + 与当前区间首元素交换

---

1.  **算法5大特性的核心意义**
    - 有穷性是算法和程序最核心的区别：无限循环的程序（如操作系统、常驻服务）不满足算法的定义。
  
    - 确定性+可行性：要求算法不能有模糊的指令，且必须是可落地执行的，比如“随机生成一个数字”满足要求，但“生成一个无法被写出的数字”不满足可行性。
  
    - 输入可以为0（比如生成固定序列的算法），但输出必须至少有1个，否则算法没有完成任务的意义。
  
2.  **选择排序的基础特性**
    这是入门级的比较排序算法，属于原地排序（无需额外的大内存空间），后续我们会通过渐近记号分析它的时间复杂度为$\Theta(n^2)$。

---

## 二、§1 算法分析的核心内容
我们分析算法，核心是评估它的**时间开销**和**空间开销**，而非和机器、编译器绑定的具体运行时间。

### 分析的前提假设
为了剥离机器差异，我们做3个基础假设：

1.  指令按顺序执行

2.  每条指令都是简单操作，执行恰好花费1个时间单位
   
3.  整数大小固定，内存无限

我们通常聚焦两个核心时间复杂度函数：

- $T_{avg}(N)$：平均情况时间复杂度，输入规模为N时的平均运行时
  
- $T_{worst}(N)$：最坏情况时间复杂度，输入规模为N时的最大运行时间
  
> 若算法有多个输入维度，复杂度函数可包含多个参数。

---

### 示例1：矩阵加法
```c
// 矩阵a + 矩阵b = 矩阵c，矩阵维度为rows行cols列
void add ( int a[][ MAX_SIZE ], 
           int b[][ MAX_SIZE ], 
           int c[][ MAX_SIZE ],
           int rows, int cols )
{
    int i, j;
    for ( i = 0; i < rows; i++ )
          for ( j = 0; j < cols; j++ )
                c[ i ][ j ] = a[ i ][ j ] + b[ i ][ j ];
}
```
精确步数计算：$T(rows, cols ) =  2 \times rows \times cols + 2 \times rows + 1$

> 问题：如果 rows 远大于 cols 该如何处理？
> 
> 回答：交换 rows 和 cols 的遍历顺序，优化内存缓存命中率。

---

### 示例2：数组求和的迭代实现
```c
float sum ( float list[], int n )
{  /* 累加数组中的所有数字 */
   float tempsum = 0; 
   int i; 
   for ( i = 0; i < n; i++ ) 
       tempsum += list[i];
   return tempsum;
}
```
精确步数计算：$T_{sum}(n) = 2n + 3$

---

### 示例3：数组求和的递归实现
```c
float rsum ( float list[], int n )
{  /* 递归累加数组中的所有数字 */
   if ( n )    
       return rsum(list, n-1) + list[n - 1];
   return 0;
}
```
精确步数计算：$T_{rsum}(n) = 2n + 2$

> 注意：虽然递归实现的步数公式数值更小，但每一步的实际执行耗时远高于迭代。

---

> 我们真的有必要精确计算每一步的执行次数吗？

> 结论：精确计数非常繁琐，且在不同机器、编译器下无普适性。我们真正需要关注的，是**运行时间随输入规模增长的趋势**。

---

1.  **为什么不关注精确步数，只关注增长趋势？**
    精确步数的常数系数会受机器性能、编译器优化、编程语言影响，比如同一段C代码和Python代码的单步执行耗时天差地别，但它们的增长趋势是完全一致的。算法分析的核心是对比算法本身的优劣，而非硬件/环境的差异，因此增长趋势才是核心。
2.  **平均情况 vs 最坏情况**
    我们日常分析算法时，**绝大多数场景都聚焦最坏情况复杂度**：它给出了算法运行时间的上界，能保证算法在任何场景下都不会比这个更慢；而平均情况需要统计所有输入的概率分布，计算复杂，且很多场景下和最坏情况的增长趋势一致。
3.  **迭代与递归的实际差异**
    递归的步数公式看似更优，但函数调用会产生额外开销：每次递归都要创建栈帧、保存现场、传递参数、返回结果，不仅耗时更高，还可能出现栈溢出的问题。工程中，非必要场景不推荐使用递归实现简单循环逻辑。
4.  **矩阵加法的遍历优化原理**
    C语言中二维数组是**行优先存储**的，按行遍历能连续访问内存，缓存命中率极高；按列遍历会频繁跳变内存地址，缓存频繁失效，性能会大幅下降。因此当行数远大于列数时，交换遍历顺序能显著提升运行效率。

## 三、§2 渐近记号（O、Ω、Θ、o）
我们统计步数的核心目的：预测运行时间随输入规模N的增长趋势，从而对比两个程序的时间复杂度。因此我们真正需要关注的，是时间函数$T_p$的**渐近行为**。

举个例子：假设 $T_{p1}(N) = c_1N^2 + c_2N$，$T_{p2}(N) = c_3N$。哪个更快？

无论$c_1、c_2、c_3$的取值如何，总会存在一个阈值$n_0$，当所有 $N > n_0$ 时，都有 $T_{p1}(N) > T_{p2}(N)$。

> 结论：只要知道$T_{p1}$的增长量级约为$N^2$，$T_{p2}$约为$N$，那么当输入规模N足够大时，$P_2$一定更快。

---

### 2.1 渐近记号的定义
1.  **大O记号（上界）**
    【定义】若存在正常数$c$和$n_0$，使得对于所有 $N ≥ n_0$，都有 $T(N) ≤ c·f(N)$，则记 $T(N) = O(f(N))$。

    > $T(N)$的增长速度**不会超过**$f(N)$，$f(N)$是$T(N)$的时间上界，对应算法的最坏情况上限。

2.  **大Ω记号（下界）**
    【定义】若存在正常数$c$和$n_0$，使得对于所有 $N ≥ n_0$，都有 $T(N) ≥ c·g(N)$，则记 $T(N) = Ω(g(N))$。

    > $T(N)$的增长速度**不会低于**$g(N)$，$g(N)$是$T(N)$的时间下界，用于描述算法的理论性能下限。

3.  **Θ记号（紧界）**
    【定义】当且仅当 $T(N) = O(h(N))$ 且 $T(N) = Ω(h(N))$ 时，记 $T(N) = Θ(h(N))$。

    > $T(N)$的增长速度**与$h(N)$完全相当**，$h(N)$是$T(N)$的紧确界，说明算法的上下界一致。

4.  **小o记号（非紧上界）**
    【定义】若 $T(N) = O(p(N))$ 且 $T(N) ≠ Θ(p(N))$，则记 $T(N) = o(p(N))$。

    > $T(N)$的增长速度**严格慢于**$p(N)$，是不紧的上界。

---

### 2.2 使用规范
- 对于大O记号，$2N + 3 = O(N) = O(N^{k≥1}) = O(2^N) = ...$，我们**始终取最小的$f(N)$**，也就是最紧的上界。
  
- 对于大Ω记号，$2N + N^2 = Ω(2^N) = Ω(N^2) = Ω(N) = Ω(1) = ...$，我们**始终取最大的$g(N)$**，也就是最紧的下界。

---

### 2.3 渐近记号的运算规则
1.  若 $T_1(N) = O(f(N))$ 且 $T_2(N) = O(g(N))$，则：
   
    - (a) $T_1(N) + T_2(N) = max( O(f(N)), O(g(N)) )$（加法取最高阶，低阶项可忽略）
  
    - (b) $T_1(N) * T_2(N) = O( f(N) * g(N) )$（乘法阶数相加）
2.  若 $T(N)$ 是一个k次多项式，则 $T(N) = Θ(N^k)$。
3.  对于任意常数k，都有 $log^k N = O(N)$。这说明**对数函数的增长速度极其缓慢**，几乎接近常数级。

> 重要注意事项：渐近对比的前提是输入规模N**足够大**。
> 
> 例如：$T_{p1}(N) = 10^6 N$，$T_{p2}(N) = N^2$。虽然$Θ(N^2)$的增长速度比$Θ(N)$快，但当 $N < 10^6$ 时，$P_2$仍然比$P_1$更快。

---

### 2.4 常见复杂度的增长速度排序
从慢到快（性能从优到差）依次为：
$log\ n$ < $n$ < $n\ log\ n$ < $n^2$ < $2^n$

---

### 2.5 时间复杂度分析的通用规则
1.  **for循环**：for循环的运行时间，最多为循环内语句（包括循环判断）的运行时间 × 循环迭代次数。
2.  **嵌套for循环**：嵌套循环内语句的总运行时间，为语句的运行时间 × 所有循环的规模乘积。
3.  **连续语句**：运行时间相加（最终取最大值，即最高阶项）。
4.  **IF/ELSE语句**：对于代码片段
    ```c
    if ( Condition )  S1;
    else  S2;
    ```
    运行时间不超过「条件判断的运行时间 + S1和S2中运行时间的较大值」。
5.  **递归**：递归的时间复杂度需通过递推式分析。

---

### 示例1：矩阵加法的渐近复杂度分析
```c
void add ( int a[][ MAX_SIZE ], 
           int b[][ MAX_SIZE ], 
           int c[][ MAX_SIZE ],
           int rows, int cols )
{
    int i, j;
    for ( i = 0; i < rows; i++ )
          for ( j = 0; j < cols; j++ )
                c[ i ][ j ] = a[ i ][ j ] + b[ i ][ j ];
}
```
- 外层循环：$Θ(rows)$
  
- 内层循环+赋值语句：$Θ(rows × cols)$
  
- 整体渐近复杂度：$T(rows, cols) = Θ(rows × cols)$

---

### 示例2：斐波那契数列的递归实现
**递推定义**：$Fib(0) = Fib(1) = 1$，$Fib(n) = Fib(n-1) + Fib(n-2)$
```c
long int Fib ( int N )
{
	if ( N <= 1 )
	    return 1;
	else
	    return Fib( N - 1 ) + Fib( N - 2 );
}
```
#### 复杂度分析
递推式：$T(N) = T(N-1) + T(N-2) + 2$
可通过数学归纳法证明：$T(N) ≥ Fib(N)$，$T(N)$呈**指数级增长**。

> 为什么这个递归实现的效率极差？
> 
> 根本原因：存在大量的重复计算，比如$Fib(5)$会计算$Fib(4)$和$Fib(3)$，$Fib(4)$又会重复计算$Fib(3)$，随着N增大，重复计算量爆炸式增长。

---


1.  **四大渐近记号的核心使用场景**
    - 日常开发、算法题解中，我们说的“时间复杂度O(n)”，默认指**最坏情况的紧上界**，也就是最贴合实际的Θ记号，只是行业内习惯用O记号代指。
    - Ω记号多用于算法理论证明，比如“基于比较的排序算法，时间下界为Ω(n log n)”，用来证明某类算法的性能天花板。
    - 小o记号多用于严格的数学分析，日常开发极少使用。
2.  **为什么低阶项和常数系数可以忽略？**
    当输入规模N足够大时，最高阶项完全决定了增长趋势。比如$10000n + 1000000$和$0.0001n^2$，当n超过1亿时，后者的增长速度会远超前者。算法分析的核心是“大规模输入下的性能表现”，因此常数系数和低阶项可以忽略。
3.  **对数复杂度的优势**
    对数级复杂度$O(log\ n)$是极其优秀的性能，比如二分查找的时间复杂度就是$O(log\ n)$。哪怕n是2的30次方（约10亿），$log_2 n$也只有30，几乎和常数级$O(1)$的性能无差别。
4.  **递归复杂度的分析要点**
    斐波那契的递归实现是经典的“反面教材”，它的时间复杂度是指数级$O(φ^n)$（φ为黄金分割比≈1.618），n到40左右就会出现明显的卡顿，n到100几乎无法计算。优化方案包括：迭代实现、记忆化递归（动态规划）、矩阵快速幂、通项公式等，能把时间复杂度降到$O(n)$甚至$O(log\ n)$。
